# Replay Regression Fixtures

These `.gesturesample` files are **hand-authored test fixtures**, not real user
recordings. Do not confuse them with runtime samples under
`~/Library/Application Support/GestureFire/samples/`.

## Purpose

`MultiRecognizerReplayTests` loads these files via `Bundle.module` and feeds
them through `RecognitionLoop.replay()` to guarantee that recognition results
for each fixture stay identical across every Phase 3 step. This is the Phase 3
replay invariant: adding a new recognizer is never an excuse for an older one
to regress. If a prior fixture goes red, roll back before continuing.

## Structure

```
tiptap/                 — TipTap 4-direction baseline (Step 0)
corner/                 — Corner taps (added with CornerTapRecognizer)
multitap/               — Multi-finger taps (added with MultiFingerTapRecognizer)
multiswipe/             — Multi-finger swipes (added with MultiFingerSwipeRecognizer)
```

Each subdirectory holds one fixture per gesture variant, encoded as JSONL using
`GestureSample.toJSONL()`.

## Regeneration

Fixtures are generated by the env-gated test `FixtureGenerator` in the
`GestureFireRecognitionTests` target. To regenerate:

```bash
GFIRE_GENERATE_FIXTURES=1 ./scripts/test.sh --filter FixtureGenerator
```

The generator writes to the source directory (not the build bundle). Review
the diff, then commit the regenerated files. Always run the full replay suite
after regeneration to confirm every asset still produces its expected gesture.
